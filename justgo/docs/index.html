01 / START HERE

What JustGo actually is.

JustGo is an open-source SwiftUI app for planning metro journeys in China. Its catalog, route-picker baselines, station packs, official-resource hyperlinks, and live requests are separate layers. A city appearing in the catalog does not mean every layer exists. JustGo is not a transit operator, a live signaling system, or a guarantee that a station facility will be working.

The short version

Included files work as app content. Operator pages and files open only after a rider tap and are not bundled, fetched, previewed, cached, prefetched, or redistributed. Hong Kong live arrivals require a government transport API request. Personal station media stays entirely on-device.

02 / ROUTE PIPELINE

From two places to one explained route.

A route is assembled in layers. The source of each layer stays visible so a missing schedule cannot be mistaken for a routing failure, and an estimated exit cannot be mistaken for an official instruction.

01Resolve places

Typed POIs and addresses are searched with MKLocalSearch. “Current Location” asks Core Location only when the rider invokes it. Saved and favorite places may already contain coordinates.

02Choose a network

Both endpoints must lie within 25 km of a bundled network’s bounds. If more than one network qualifies, JustGo chooses the one with the smallest combined endpoint-to-network access distance.

03Choose access stations

For each endpoint, the four nearest stations become graph candidates. Their walking distance is included in the path cost rather than forcing the single nearest station.

04Search the metro graph

A minimum-cost graph search runs three times. Fastest uses normal costs; fewest transfers raises the transfer penalty from 300 to 1,200 seconds; least walking triples the access-walking weight.

05Build the legs

Train time is estimated as max(60 s, distance ÷ 9.7 m/s + 30 s) per graph edge. Transfers add five minutes. Apple walking directions are requested for the first and last leg.

06Fall back honestly

If walking directions fail, distance becomes straight-line distance and duration uses 1.25 m/s. The route receives a “walking distance is estimated” note instead of silently presenting the fallback as mapped walking.

07Enrich only from available fields

Boarding, transfer, and arrival stations are checked against the included city pack. Missing fields remain unknown. An official-resource hyperlink is tap-to-open URL metadata, not included map content.

08Rank and explain

The selected route mode reranks the unique alternatives. Feasibility, service-hours status, comfort, source coverage, and personal reports are then folded into the confidence explanation.

Network requirement

The metro graph itself is bundled, but fresh place search and Apple walking routes require connectivity. Existing saved coordinates can reduce that dependency; they do not turn the entire planning flow into a guaranteed offline service.

04 / CONFIDENCE SCORE

A score with receipts.

Every route starts at 100. The current service applies fixed additions and deductions, clamps the result to 0–100, then labels 80–100 high, 60–79 medium, and 0–59 low.

100 − 8 × transfers − min(20, floor(walking metres ÷ 100)) − 10 if schedules are not official − 10 if a source-backed layout reference is unavailable − 10 if accessibility is not official − min(15, 5 × missing core data categories) + feasibility adjustment + service-hours adjustment + comfort adjustment + alternative / full-coverage bonuses = clamp(0…100)

Service ended / not started / last train soon−40 / −25 / −12Derived from static first/last schedule windows for the boarding line.

Not a probability

An 84 does not mean an 84% chance of success. The score is a readable comparison heuristic over the data currently available to the app. Station conditions and operations can change after the data was captured.

07 / ACCESSIBILITY

Support, with honest boundaries.

Accessibility affects the profile, route ordering, warnings, station detail, and the Go companion. JustGo distinguishes an affirmative facility record from the absence of evidence.

Route assessment states

StateCurrent rule
ConfirmedEvery critical stop has accessibility data and every record marks the station fully accessible.
LikelyEvery critical stop has accessibility data and each has either an elevator or a wheelchair ramp.
Barrier detectedApple walking directions or route warnings mention stairs.
UnknownCoverage is incomplete or does not meet a stronger rule. Unknown is not the same as inaccessible.

Rider controls

  • Wheelchair access, elevator preference, stair avoidance, and maximum walking distance shape route filters and ranking.
  • Audio navigation speaks each Go step; vibration alerts add haptic feedback; visual announcements show a temporary step banner.
  • Simplified UI removes nonessential planner sections; step-by-step guidance can open the Go companion automatically.
  • Accessibility reports are personal notes stored locally. They can lower feasibility and confidence for matching routes, but are not sent to other riders.
Step-free is not guaranteed

Only Hong Kong has included accessibility records, covering 98 of 162 matched stations. Beijing has none. Verified indoor transfer coverage is zero, so the app cannot prove a complete step-free indoor path or a working elevator.

08 / GO COMPANION

Trip steps without indoor-routing claims.

Go converts a chosen route into ordered walk, ride, transfer, and arrival cards. The rider advances and reverses steps manually.

  • Walking steps can show an Apple Maps route preview when path geometry is available.
  • Verified indoor transfer coverage is zero; route-picker geometry and external maps do not establish an indoor path, platform, door, or transfer.
  • A Hong Kong map link opens its operator destination only after a rider tap. JustGo does not bundle, fetch, preview, cache, prefetch, or redistribute that destination.
  • Speech, haptics, and visual banners follow the rider’s local accessibility preferences.
  • Get-off reminders use the estimated segment duration minus the chosen lead time. Both the in-app banner and local notification explicitly say they are time estimates, not a live train position.
  • Hong Kong live-arrival boards are a separate online feature that contacts the government transport API; they do not provide indoor positioning.
  • The active route is saved locally when Go starts so the planner can offer to resume after the app is terminated.

10 / KNOWN LIMITS

What the labels do not promise.

  • No universal live coverage: only Hong Kong has live-arrival references, covering its 162 matched stations; a live request is not vehicle positioning or an offline schedule.
  • No included static schedules: neither Beijing nor Hong Kong has a static schedule dataset in the verified snapshot.
  • No facility uptime guarantee: Hong Kong accessibility records describe sourced facts, not current operating status.
  • No verified indoor transfer data: coverage is zero; route-picker geometry and linked maps do not prove an indoor path, platform, door, or transfer.
  • At most three unique graph alternatives: several user-facing modes rerank that set rather than generating a bespoke path.
  • Walking text is language-dependent: stair detection relies in part on Apple walking instructions and source text.
  • Official-resource links are not app content: 330 directory records resolve to 275 unique external targets and open only after a rider tap.
  • Source-pending means incomplete: 44 cities have geometry only and 12 are catalog-only.

11 / DEVELOPMENT

Build the current source.

The repository is MIT-licensed and built with SwiftUI plus Apple frameworks. Consult the repository for current build requirements and configuration.

  1. Clone E-Rail/JustGo.
  2. Read the repository README and project settings for the current Xcode and platform requirements.
  3. Keep the included, external-reference, live-API, and personal-media categories separate when adding data.
  4. Add required source and media acknowledgments to the attribution record.

Architecture in one sentence

SwiftUI views and observable view models call injected service objects; network and graph services are actor-isolated where they own mutable caches; route and station facts remain plain Codable model values.
